Objective

This is a project for Multiverse. The objective was to create an API with a login.

Criteria:

  1. MVP Application and REST: A user should be able to log in to a deployed app and create, read update and delete from the database by accessing RESTful routes.
  2. Login and Hashing: A user should be able to login to my API and any saved passwords are hashed and salted before saved to the database. (note: If you use OAuth, you might not even store passwords at all.)
  3. Register: A user should be able to sign up for the API and be granted authorization to access the API.
  4. Authorization: A users API shouldbe protected from unauthorized users. An unauthorized user should see a message saying they do not have access to the API. A user should not be able to create new entities without logging in. A user data should only be accessible, editable, and deleteable by the user.
  5. Associated Data: A user should be able to read a single entry, see the associated user info and other associated data.
  6. Admin vs. User: An admin should have special super-user account type that allows access to content users don't have access to. A basic user should only see their own entries. An admin should be able to see all entries and regardless of user and edit other users information via the API.

Approach

For this project I used Auth0 for the login as opposed to saving and hashing paswords. I used Auth0s log in with Google feature. My API checks the SQLite3 database for an email that matches the one you logged in with and displays your profile information and will only allow you to see entries associated with that email.

Steps Taken

I started this project by looking for examples of other APIs and looking at the work we had done in the weeks prior, because even though we spent a few weeks learning about them, I still felt like I didn't understand all the things that go into creating one. I don't think I had any examples of my own that worked completely. All the little projects we had done with APIs, I could only get to 'kind of' work. I started with doing the parts that I was confident that I could do correctly. I picked journal entries for my API because I didn't want to overcomplicate things knowing I really didn't know what I was doing yet. The first thing I did was create the Sequelize models because I knew what I would need to include in them, and then I started working on the database. I think I had used the wrong database at first and I had to start over a few times. I created a file with seed data and then started working on seeding the database. After that I started with the routes. I tested themwith Postman and then I added Auth0 right after. After I was done with everything I tested the Sequelize models using Jest.

Challenges

I struggled with this project because I did everything on my own, so I spent a lot of time troubleshooting. Getting Auth0 set up was tricky and it took up the majority of the time I spent working on this. Another thing that was a challenge for me was getting the SQLite3 database set up. My database either kept dropping the tables or duplicating everything over and over. This is something I struggled with for a while. I had to find examples of similar projects to reference to figure out how to make it work the way I intended. Auth0 took a while for me to set up as well and was more difficult than I expected.
